Loop, the tutorial

Stop tuning prompts. Start editing the loop. Hands-on? → Workshop 🛠️ · Or play → Loop Lab 🎮

Loop is a small natural-language DSL for loop engineering. A .loop file describes a self-correcting, human-gated coding workflow — its objective, the context it may read, the actions it may take, how it verifies itself, and when it stops. This page teaches the whole language from the first line to full A-to-Z pipelines, every section grounded in a real example you can run.

a claude session● ● ●
…or one loop
# run in chat:  /loop run rate-limit.loop
# (the agent knows Loop from AGENTS.md — your repo's memory)

git:
  work on a branch
  commit when the goal is met
  open a pull request

loop "add API rate limiting":
  goal: requests are rate-limited per API key
  done when "pnpm test rate-limit" passes
  look at: the API and its middleware, and the last failure
  allow edits automatically, but ask me before pushes
  each cycle: plan, then act, then observe
  also: run a security check
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "thrashing"
Skill: /loop runs it · Memory: AGENTS.md + reflect · Git: branch + commit + PR, never main

Left: a dozen messages, and it still pushed to main. Right: the whole job, written once — it runs, verifies, and stops only when it's really done.

1 · What is a loop basic

AI writes the code now — but you are still the conductor. Every coding task is really five decisions:

DecisionIn a .loopQuestion it answers
Objectivegoal:What are we trying to do?
Contextlook at:What may the agent read first?
Actionsallow… / ask me before…What may it do, and what needs a human?
Verificationdone whenHow do we know it worked?
Stoppingwhen… / after N triesWhen do we stop — done, or thrashing?

Here are all five decisions as one real loop — every line is one of the rows above:

loop "fix the failing test":                       # the work
  goal: the cart total is correct with a coupon    # Objective
  look at: the checkout code, and the last failure      # Context
  allow edits automatically, ask me before pushes  # Actions
  done when the cart coupon tests pass # Verification
  after 6 tries: stop and warn "stuck"             # Stopping

When the model took over the building, those five decisions got buried in prompts. Loop promotes them to first-class, editable knobs. At runtime they drive five phases — plan → act → observe → reflect → stop — the diagram above. You don't re-tune a prompt; you edit the loop.

The two ideas to keep
  • Edit the loop, not the prompt. The control structure is the artifact.
  • You can't fake done. done when runs a real command — a test, a scanner, a script. The loop stops only when the world agrees.

You're already doing loop engineering — by hand why

Fix a bug with Claude today and you already run a loop — in your head. You are the for-loop, the exit condition, and the memory:

today — you run the loop
plan — you describe the change
Claude edits the files
observe — you run the test
↳ it failed
reflect — you read the error, work out what broke
↺ you re-prompt with a fix… then again… and again
— and you decide when it's "done",
— you remember not to push to main,
— you babysit every cycle. At 2am. Badly.
with Loop — the loop runs itself
loop "fix the bug":
  goal: the failing checkout tax test passes
  done when the checkout tax tests pass
  look at: the checkout code, and the last failure
  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "stuck"
Never forgets the test, the thrash guard, or "don't push to main." Reflects every cycle. Reusable + shareable.
The shift
  • Today you ARE the loop — the runtime, the exit condition, the memory. It's on you every iteration, and none of it is written down or repeatable.
  • With Loop you write the loop once — the runtime runs it. You're promoted from runtime to author: you still watch and answer the gates, you just stop being the for-loop.

Prompt vs Loop — why not just prompt? why

You could just say "fix the bug." So why write a loop? A prompt fires once and trusts the model's word that it's done. A loop verifies, self-corrects, and stops only when the work is provably finished.

Just promptingA loop
"Done" meansthe model says "done"a real command passes — done when "…"
On failureyou notice, re-prompt, repeatreflects on the failure, re-plans automatically
Stopswhen the model stops typingwhen the check is green — or warns after N tries
Risky actionshope it asks firstgated; never pushes to main/master
Scopewanders the codebaselook at: keeps it in your module
Repeatablere-type it, get driftre-run the same file, same shape
Shareablea paragraph in Slacka .loop in the repo, reviewable in a PR

Same task, both ways

The prompt:

"Fix the failing checkout tax test in src/checkout and make sure nothing else breaks."
→ the agent edits, replies "Done — fixed the rounding." Did checkout.spec.ts::tax actually pass? The whole suite? You re-run it yourself. Failed? Re-prompt. (And it may have run git push on the way.)

The loop:

loop "fix the checkout tax test":
  goal: the checkout tax test passes with no regressions
  done when the checkout tax tests pass
  look at: the checkout code, and the last failure
  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "tax fix thrashing"

Runs the test every cycle. Fails → reflects on why → fixes again. Stops only when the test is green (or warns after 6). Works on a branch, never touches main. The same file works tomorrow, and a teammate can read exactly what "done" meant.

The point
  • Prompting asks for an answer. A loop guarantees a result.

When to reach for Loop — by scope orientation

Loop earns its keep when work has a verifiable "done" and benefits from self-correction or human gating. The bigger and riskier the task, the more structure you reach for. Match the construct to the unit of verifiable done:

ScopeReach forBecause
Small code patch
typo, rename, one-liner		Usually nothing — just edit. A bare loop only if a check must stay green.For a trivial edit the loop overhead costs more than it saves.
Bug fixA single loop with done when <the failing test> passes + reflect + a thrash guard.The reflect back-edge is built to turn a red test green, learning from each miss.
FeatureA pipeline (one story, several stages) — or a flow of story files.A feature has several concerns in order — build, tests, security, UX — each its own gate.
Full app / epicAn A-to-Z flow: discover → design → for each story.The stories aren't known yet; discovery produces them, then one checklist runs per story.

Recipe — a small code patch

For a pure typo or rename, skip Loop and just make the edit. Reach for a tiny loop only when a check must keep passing — e.g. a dependency bump that must not break the suite:

loop "bump lodash, keep tests green":
  goal: upgrade lodash to ^4.17.21 with no regressions
  allow edits automatically
  each cycle: act, then observe
  done when "pnpm test" passes
  after 3 tries: stop and warn "upgrade breaks tests — needs a human"

Use when: a mechanical change guarded by a command. No plan step needed — just act + observe.

Recipe — a bug fix

The bread-and-butter loop. Point it at a failing test (write one first if there isn't one), scope the files, and let the reflect edge work:

loop "fix: cart total wrong with a coupon":
  goal: the cart total is correct when a percentage coupon is applied
  done when the cart coupon tests pass
  look at: the cart total and coupon code, and the last failure
  allow edits automatically, but ask me before migrations
  each cycle: plan, then act, then observe
  when it fails: reflect on which layer broke, then plan again
  after 6 tries: stop and warn "thrashing — likely a spec ambiguity"

Use when: a reproducible bug with a check that proves the fix. The warning is your cue that the spec itself is unclear.

Recipe — a feature

A feature bundles several concerns that should each pass before you ship. Model it as a pipeline of stages — implement, then no-regressions, then security, then a human UX review:

pipeline "feature: CSV export on the reports page":

  stage build:
    goal: a user can download the current report as CSV
    look at: src/reports/, src/export/, and the last failure
    allow edits automatically, but ask me before migrations or pushes
    each cycle: plan, then act, then observe
    done when "pnpm test reports/export" passes
    when it fails: reflect on which layer broke, then plan again
    after 8 tries: stop and warn "export story stuck"

  stage regression:
    goal: the full suite stays green
    each cycle: act, then observe
    done when "pnpm test" passes

  stage security:
    goal: no high or critical vulnerabilities introduced
    each cycle: plan, then act, then observe
    done when "semgrep --severity=high" finds nothing

  stage ui:
    goal: the download button matches design and is keyboard-accessible
    each cycle: plan, then act, then observe
    a human reviews before stopping

Use when: one self-contained feature with multiple gates. If it spans several independent stories you want as reusable files, use a flow of .loop files instead (Section 11).

Recipe — a full app / epic

For a large build where the stories aren't known up front, go A-to-Z. An interactive discovery interviews you and writes the plan, design gets approved, then the per-story checklist runs for every story — pausing to ask you when one fails:

flow "epic: subscription billing":
  run "discover.loop"          # interview the team -> write sprint.yaml
  then run "design.loop"       # data model + API design; a human approves
  then for each story in "sprint.yaml":
    run "story-template.loop"   # plan, build, test, scan — once per story

The discovery step is just a loop — it runs BMAD discovery (interviews you), then writes the plan. It's done when the artifact exists, not when the chat goes quiet:

loop "discover":
  goal: run BMAD discovery — interview the team and write sprint.yaml (the epic's stories)
  look at: docs, and the last failure
  allow edits automatically
  each cycle: plan, then act, then observe
  done when "test -s sprint.yaml" passes
examples/bmad/atoz/discover.loop — runs in conversation; ends on the sprint.yaml artifact check.

Use when: an epic or a from-scratch module. Run it in conversation (/loop run) so discovery can interview you (Section 13).

Rule of thumb
  • One check → a loop.
  • Several ordered checks → a pipeline (one file) or a flow (many files).
  • One check repeated over a discovered list → for each.
Skip Loop when: the task has no verifiable "done" (a pure spike/exploration), it's a trivial edit you'd finish faster by hand, or you can't yet state the goal — in that last case, do a discovery loop first to produce the spec, then loop the build.

Getting started setup

That's the why. From here down is the deep dive — set Loop up once, then learn the language line by line.

Two ways to run Loop: inside a Claude Code chat (the bundled skill — recommended), or the CLI — and a VS Code extension to write loops by hand and ▶-run them. Two one-time setup steps, then your first loop:

1 · Install the skill

Copy the bundled Claude Code skill into your skills folder — then /loop works in any repo:

cp -r .claude/skills/loop ~/.claude/skills/

2 · Give the agent its memory — AGENTS.md

Drop Loop's AGENTS.md (the full language reference) into your repo. It travels with the project, so any agent that opens the repo already knows the Loop language — the project's persistent memory of how to write a .loop. Methods are shared the same way: use the <X> method pulls in a .loop preset another repo can reuse.

3 · Run your first loop

/loop fix the failing auth test in src/auth, gate any database migration   # writes a .loop
/loop run examples/fix_test.loop                                          # runs it, in the chat

Describe the work and the skill writes the .loop; name a file and it runs the loop natively in the session — you watch every plan → act → observe → reflect step and answer human gates right in the chat. Prefer a terminal? The same files run headless via the CLI — see Running a loop.

That first command writes a file like this — yours to read, edit, and re-run:

# /loop "fix the failing auth test in src/auth, gate any database migration" writes:
loop "fix the failing auth test":
  goal: the auth suite passes in src/auth
  done when "pnpm test src/auth" passes
  look at: the auth code, and the last failure
  ask me before migrations
  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "stuck"

4 · Edit in VS Code optional

A .loop is just text — let the agent draft it, or write one by hand. The Loop VS Code extension makes hand-authoring first-class:

  • Syntax highlighting for the whole language — goal, done when, gates, flow, for each, git:.
  • Completion + formatting — keyword snippets so you type a loop from muscle memory, not a cheat sheet.
  • A ▶ Play button on every loop (the Loop: Run this loop CodeLens) — run the file you're editing without leaving the editor.
  • New from template (command palette → Loop: New from template) — scaffold a ready-to-edit .loop bundle.

So the two authoring paths are interchangeable: prompt the agent to generate a loop, or open VS Code and write the .loop yourself — same file, same engine.

Safe from the first run: with no git: block, Loop works on a branch and commits when the goal is met, and never pushes to main/master — see Git strategy.

2 · Your first loop basic

The smallest useful loop: keep working until a test passes.

loop "fix test":
  goal: the checkout tax test passes
  done when the checkout tax tests pass

  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
examples/fix_test.loop

Read it top to bottom:

1loop "fix test": — names a self-correcting loop. Indentation is structural (two spaces); loop sits at column 0; # starts a comment.

2goal: — the objective, in plain language.

3done when the test "…" passes — the verification. The runtime actually runs that test each cycle.

4each cycle: — the repeated steps. plan, then act, then observe is the default, so you can omit it.

5when it fails: reflect, then plan again — the feedback edge: on a failed check, turn the failure into context and re-plan. That's what makes it self-correcting.

Run it: in a chat with the Loop skill, say /loop run examples/fix_test.loop and the assistant runs the cycle right there. Or headless: loop run examples/fix_test.loop.

3 · done when — how a loop verifies itself basic

The predicate is the spine of the whole idea. Four forms:

done when the test "billing.spec.ts::apostrophe" passes   # a named test (runs via your test runner)
done when "pnpm test" passes                               # a shell command, success = exit 0
done when "semgrep --severity=high" finds nothing          # exit 0 AND empty output
done when a human confirms "the UI looks right"            # a person is the check
Why this matters
  • A predicate is a real command run with your privileges — like an npm script or a Makefile target. So treat a .loop from an untrusted source as you would their shell scripts.
  • finds nothing is how you say "this scanner must report zero" — it requires both exit 0 and empty output.
  • The a human confirms "…" form is decided by a person — it's satisfied when you run the loop in conversation; the headless shell verifier returns human check required: … and never passes on its own.
  • A loop with no done when has no machine check, so it must finish through a human path — a review gate, an approved plan-first pass (a plan-only loop), or an explicit when …: stop — otherwise it runs to the hard cap. Always give it a real check when one exists.

4 · look at + action policy basic

Scope what the agent reads, and decide what it may do on its own versus what needs your sign-off.

look at: the billing form, the settings API, and the last failure
allow edits automatically, but ask me before migrations or pushes

look at: points the agent at the context that matters. Items can be exact file paths or plain-language descriptions — the agent locates the actual files from descriptions before planning, so you don't need to know the precise path. The special phrase and the last failure feeds the previous failure's output back in.

The policy line splits actions into auto and confirm classes. The canonical classes are edit, migrate, push, deploy, delete (plurals/synonyms normalize). allow … automatically grants without asking; ask me before … pauses for a one-time confirmation that's remembered for the run.

Map to tools: the runner translates classes into tool grants — edit → Edit/Write, push → git push, and so on. Read tools are always allowed.

5 · The cycle and the reflect back-edge intermediate

each cycle: lists the steps, in order — any subset of plan, act, observe:

each cycle: plan, then act, then observe   # full self-correcting unit
each cycle: act, then observe              # skip planning — just do + check
  • plan — read the look at: files, decide the smallest change toward the goal. (Runs read-only.)
  • act — make the change, honoring the policy.
  • observe — run the done when check and read pass/fail.

On a failed observe, when it fails: reflect, then plan again fires. reflect reads the failure output and writes a short diagnosis; that diagnosis becomes context for the next plan. This is the back-edge — the orange arc in the diagram — and it's the difference between an agent that retries blindly and one that learns from each miss.

Safety net: regardless of your rules, the engine has an absolute hard cap of 25 cycles per loop, so a loop can never spin forever.

6 · Transitions and the thrash guard intermediate

when …: rules run after each observe. The conditions and actions:

when it passes and the goal is met: stop
when it fails:                      reflect on which layer broke, then plan again
when blocked:                       ask a human
after 6 tries:                      stop and warn "thrashing"
ConditionsActions
it passes · it passes and the goal is met · it fails · blocked stop · stop and warn "…" · reflect [on <focus>] · plan · act · observe · ask a human
Precedence is fixed, not source-order
  • The engine evaluates transitions in the order blocked > attempts > pass > fail.
  • So after N tries: stop and warn — the thrash guard — always wins once the threshold is crossed, even if a when it fails rule is listed first. Put a thrash guard on every self-correcting loop; the warning is your signal that it got stuck.

· Model policy — cheap where cheap is enough intermediate

A loop is several LLM calls per cycle, so naively it could cost more than one prompt. The models: policy fixes that: name two tiers and the engine runs the cheap-thinking phases on the fast one and the hard act on the strong one — a stack of cheap calls plus one strong call, not N expensive prompts.

models: fast haiku, strong opus

Built-in mapping: plan, reflect and finishing also passes → fast; act → strong; observe is a shell check (no model). Override a phase or a whole scope, and it cascades file→loop→stage like the git: block:

models: fast haiku, strong opus, plan strong   # flip one phase
models: all strong                              # whole scope on one tier

--model X (CLI) or loop.model (VS Code) is the kill switch — it forces every phase onto one model. At the end of a run Loop prints a per-tier call summary, so the spend is measurable, not a guess.

7 · also — finishing passes intermediate

Lightweight passes that run only after the goal is met, in order — and are skipped if the loop failed:

also: polish the code, run a security check, update the docs

Use also for the chores that should happen on success but aren't part of the core verification. If a finishing pass needs its own pass/fail check, promote it to a real stage (next sections) instead.

8 · Human gates human-in-the-loop

Loop has five places a person steps in. The first two go inside a loop; the third is a stage gate; the fourth is a transition; the fifth is the per-run confirm prompt from the action policy (Section 4) — ask me before …, asked once and remembered.

a human approves the plan first      # approve the plan before any acting
a human reviews before stopping      # judge the result before the loop may stop
a human approves before provisioning # a hard, blocking gate (used on a stage)
when blocked: ask a human            # unblock when the agent is stuck
Where to use which
  • a human approves the plan first — high-stakes work where the plan must be right before touching anything.
  • a human reviews before stopping — subjective "looks right" goals (UI, copy) where no command can decide done.
  • a human approves before <X> — a blocking gate before a whole stage runs (deploys, provisioning).

9 · A full-power loop intermediate

Everything so far, in one real loop — the five decisions plus transitions, a thrash guard, finishing passes, and a confirm policy:

loop "fix billing apostrophe bug":
  goal: settings save when the company name has an apostrophe
  done when the test "billing.spec.ts::apostrophe" passes

  look at: billing/form.tsx, api/settings.ts, schema/settings.ts, and the last failure
  allow edits automatically, but ask me before migrations or pushes

  each cycle: plan, then act, then observe
  also: polish the code, run a security check, update the docs
  when it passes and the goal is met: stop
  when it fails: reflect on which layer broke, then plan again
  when blocked: ask a human
  after 6 tries: stop and warn "thrashing"
examples/billing_apostrophe.loop — read this one out loud; it's a design, not a prompt.

10 · Pipelines — epics and stories intermediate

A pipeline runs stages in order; a failing stage halts the rest. The natural mapping: an epic → a pipeline, each story → a stage, and each stage's body is just a loop.

pipeline "ship feature":

  stage security:
    goal: no high or critical vulnerabilities
    done when "semgrep --severity=high" finds nothing
    each cycle: plan, then act, then observe
    when it fails: reflect, then plan again

  stage build:
    goal: feature works and tests pass
    a human approves the plan first
    each cycle: act, then observe
    done when "pnpm test" passes

  stage ui:
    goal: matches design, responsive at 375px
    each cycle: plan, then act, then observe
    a human reviews before stopping

  stage deploy:
    a human approves before provisioning
    goal: infra live and healthchecks green
    done when "./scripts/health.sh" passes
    each cycle: act, then observe
examples/ship_feature.loop

Each stage carries its own verification and human gates. A real BMAD-shaped feature looks the same — analysis and architecture stages are human-gated, story stages are build loops with acceptance tests, QA gates on review:

pipeline "BMAD: user authentication":

  stage "analyze":
    goal: the auth requirements are clear and agreed
    look at: docs/brief.md
    a human approves the plan first
    each cycle: plan, then observe

  stage "story: email and password login":
    goal: users log in with email and password
    look at: src/auth/, docs/architecture.md
    done when "pnpm test auth/login" passes
    each cycle: plan, then act, then observe
    when it fails: reflect on which layer broke, then plan again
    after 8 tries: stop and warn "login story stuck"

  stage "qa":
    goal: the full suite is green and acceptance criteria are met
    done when "pnpm test" passes
    a human reviews before stopping
examples/bmad-auth.loop (abridged)

11 · flow — chaining whole files advanced

A pipeline lives in one file. A flow chains separate .loop files, run in order — so a big effort can be many small, reusable files instead of one giant one.

flow "ship":
  run "build.loop"
  then run "test.loop"
  then run "deploy.loop":
    a human approves first
examples/ship_flow.loop
flow semantics
  • Runs the whole file. run "build.loop" runs every definition in that file.
  • Fail-fast. If a file isn't satisfied, the flow halts the rest.
  • Text handoff. After each step, a short text summary of how it went is carried forward as context into the next file. It is text, not piped data structures — Loop stays a control language, not a data-flow language. Use … with the result of <step> to pull the handoff from a named earlier step instead of the previous one.
  • Cycle-safe. A file that references back to one already running errors with the chain, e.g. flow cycle: a.loop -> b.loop -> a.loop.

So an epic becomes a flow of story files, each story its own checklist pipeline:

flow "epic: authentication":
  run "story-login.loop"
  then run "story-signup.loop"
examples/bmad/epic-auth.loop

12 · for each — iterate over a plan advanced

A flow names its files up front. But what if the work items are discovered at planning time and you don't know them yet? for each is dynamic fan-out: read a plan file and run a template once per item.

flow "deliver":
  for each item in "plan.yaml":
    run "item-template.loop"
examples/foreach/deliver.loop

The source file is enumerated by type:

# plan.yaml — a list (or items under a single key)
items:
  - title: Parse the uploaded CSV into rows
    detail: accept a multipart upload, stream-parse, reject malformed input
  - title: Validate rows against the schema
    detail: required fields, types, and sane per-row error messages
examples/foreach/plan.yaml — a .md source works too: each "## " section is one item.
for each semantics
  • One run per item. The template loop runs once for every entry; that entry's text is handed in as the run's context (what to build). The checks in the template are the same for every item.
  • Source formats. .yaml/.yml → a list, or the items under a single key. .md → each ## section.
  • Halt, but you decide. If an item's checklist fails, the flow pauses and asks you: continue to the next item, or stop? Continuing accepts that item and proceeds; reaching the end means done.

The template is an ordinary pipeline — the per-item checklist, written once:

pipeline "work item":

  stage implement:
    goal: implement the work item described in the provided context
    look at: src, and the last failure
    allow edits automatically, but ask me before migrations or pushes
    each cycle: plan, then act, then observe
    done when "pnpm test" passes
    when it fails: reflect on which layer broke, then plan again
    after 6 tries: stop and warn "thrashing on this item"

  stage security:
    goal: no high or critical vulnerabilities introduced
    each cycle: plan, then act, then observe
    done when "semgrep --config auto --severity=high" finds nothing
    after 4 tries: stop and warn "security needs a human"

  stage review:
    goal: lints clean and types check
    each cycle: act, then observe
    done when "pnpm lint && pnpm typecheck" passes
examples/foreach/item-template.loop

13 · A-to-Z — discover → design → for each advanced

Put it together and a whole feature, end to end, is a single flow: an interactive discovery, a design a human approves, then the per-item checklist run for every story.

flow "epic: authentication":
  run "discover.loop"
  then run "design.loop"
  then for each story in "sprint.yaml":
    run "story-template.loop"
examples/bmad/atoz/epic.loop — run this in conversation: /loop run epic.loop

The discovery step is just a loop whose goal is to produce an artifact — here, a sprint.yaml. Run it in conversation: the assistant interviews you across as many turns as it takes (an hour-long session is fine — between your answers the loop is simply suspended). It finishes when its done when artifact check passes — not when the chat goes quiet:

loop "discover":
  goal: interview the team and capture the epic as a sprint.yaml with a stories list
  look at: docs, and the last failure
  allow edits automatically
  each cycle: plan, then act, then observe
  done when "test -s sprint.yaml" passes
examples/bmad/atoz/discover.loop
The key insight: the loop never "detects that the session ended." It re-checks a done-condition and advances the moment the output exists and validates. Discovery, design, and execution all use the same rule — done when. No new concept.

In practice — real workflows walkthroughs

You don't write Loop all day. You reach for it when a task has a clear "done." Here's how it slots into the work you already do.

A ticket from Jira (the daily driver)

You picked up PROJ-412 — "Applying a coupon can make the cart total negative." Turn the ticket into a loop:

  1. Describe it. In the chat: /loop PROJ-412: a coupon must never make the cart total negative; done when the coupon-floor tests pass — the skill writes the .loop. (Or write it yourself — the ticket's acceptance criterion becomes done when, in plain words.)
  2. What it produces:
loop "PROJ-412: coupon must not make the cart total negative":
  goal: applying a coupon never produces a negative cart total
  done when the coupon-floor tests pass

  allow edits automatically, but ask me before migrations or pushes

No failing test yet? Tell the loop to write one first — a tiny "reproduce PROJ-412 with a failing test" loop, then the fix loop. Now done when has something real to check.

A new project from scratch — plan with BMAD

Greenfield "link-shortener service." Don't hand-write the stories — let BMAD discover them, then build each one. This is the A-to-Z flow:

flow "epic: link shortener":
  run "discover.loop"
  then run "design.loop"
  then for each story in "sprint.yaml":
    run "story-template.loop"
  1. Drop AGENTS.md in the repo so the agent knows the language (see Getting started).
  2. Run it in the chat: /loop run epic.loop. Discovery interviews you (take your time — an hour is fine) and writes sprint.yaml; you approve the design; then it builds story by story, pausing to ask if one fails.
  3. Methods are presets: discover.loop uses use the BMAD method; swap BMAD for your own later. Full bundle: examples/bmad/atoz/.

Solve a bug

  1. Reproduce — get (or write) a test that fails because of the bug.
  2. Loop it — a single loop whose done when is that test, with when it fails: reflect, then plan again and an after N tries guard (the bug-fix recipe in When to reach for Loop).
  3. Run + ship — /loop run; it reflects until the test is green, commits on a branch, opens a PR if you asked.

Add a feature

  1. Model it as a pipeline — stages that each must pass: build → no-regressions → security → a human UX review (the feature recipe in When to reach for Loop).
  2. Gate the risky stage — a human approves before a deploy; a human reviews before stopping for the UI.
  3. Run it — a failing stage halts the rest, so you never ship a half-built feature.

Reuse — write a check once, run it everywhere

You'll notice the same checks repeat — e.g. "the full suite stays green":

stage regression:
  goal: the full suite stays green
  each cycle: act, then observe
  done when "pnpm test" passes

Don't copy-paste it into every pipeline. In Loop the unit of reuse is a file or a method — three ways:

  • A template, run per item. Put your standard checklist (regression, security, review…) once in story-template.loop; for each story in "sprint.yaml": run "story-template.loop" reuses those stages for every story — one edit changes the bar for all of them.
  • A standalone .loop, chained. Factor a check into its own file (regression.loop) and pull it into any flow: then run "regression.loop".
  • A method preset. Your team's "definition of done" lives in one method; use the <your-method> method brings its stages into any project — a method is just a shared .loop (see Methods are libraries).
Rule of thumb: written the same stage twice? Lift it into a template, a .loop file, or your method — then change it in one place.

15 · Git strategy intermediate

By default — with no git: block at all — Loop works on a branch and commits when the goal is met. It never pushes to main or master. This safe default applies automatically; you don't have to opt in.

A git: block lets you refine the strategy. Place it at the top of the file (before any loop definition) to apply to all loops in the file, or inside a single loop body to refine commit cadence for just that loop:

git:
  work on a branch
  commit when the goal is met
  push when done
  open a pull request

loop "add a healthcheck endpoint":
  goal: GET /healthz returns 200 with a JSON status
  done when "pnpm test health" passes
  look at: the http server and the routes module, and the last failure
  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "healthcheck stuck"
examples/git_policy.loop — opts into push + a PR; the built-in default (branch + commit, no push) would apply without the git: block.

Line forms

LineMeaning
work in placeEdit the current branch as-is.
work on a branchCreate / switch to a feature branch (the default).
work on a branch "name"Use a specific branch name.
work in a worktreeRun in an isolated git worktree.
work in a worktree "name"Named isolated worktree.
commit when the goal is metOne commit on success (the default).
commit each cycleCommit after every cycle.
commit each storyCommit after each stage in a pipeline.
commit never / do not commitNo automatic commits.
push when donePush the branch on completion.
do not pushNo push (the default).
open a pull requestOpen a PR after pushing.

Cascade

Settings resolve in three layers — each refines the one above:

  1. Built-in default — branch + commit-when-done, no push.
  2. File-level git: block — applies to all loops in the file.
  3. Per-loop git: block — refines commit cadence for that loop only.
Always-on safety — not configurable: pushing to main or master is always refused, and work in place + push when done on a protected branch is an up-front error — neither can be overridden by a git: block.

Two nearby words get confused. A Claude Code skill teaches the agent how to do Loop at all; a Loop method is a reusable .loop preset you import inside a loop.

Memory — what the loop remembers ecosystem

A loop doesn't make you re-explain context every time. It remembers at four levels — and the most important is a real file you own: AGENTS.md.

AGENTS.md — durable memory, living in your repo

AGENTS.md is a plain markdown file at your repo root that teaches any agent the Loop language (plus your own conventions). Because it lives in the repo, it travels with the project: open the repo in a brand-new session and the agent already knows how to read and write a .loop — nothing to re-paste. This is the real one that ships with Loop:

# AGENTS.md — authoring Loop (.loop) flows

This file teaches an AI assistant (Claude Code, Copilot, Cursor, …) how to
write Loop flows. When a user asks for a staged, self-correcting, or
human-gated workflow, author a .loop file using the grammar below, then run it.

## Vocabulary (the whole language)
  loop "<name>":          a self-correcting loop
  pipeline "<name>":      a sequence of stages (an epic)
  goal: <text>           what "done" means
  done when <predicate>  how the loop verifies itself
  …
AGENTS.md (abridged). Use it: drop it at your repo root. Add your team's conventions below the grammar and every loop in the repo inherits them. (CLAUDE.md is read the same way.)

The other three layers

  • Within a run — reflect. On a failed check the loop writes a short diagnosis and feeds it into the next plan — working memory, so it learns from a miss instead of repeating it.
  • Across steps — the handoff. In a flow / for each, a short text summary of each step carries forward as context; larger artifacts (a PRD, sprint.yaml) live on disk and the next step look at:s them.
  • Across the team — methods. A method preset is shared memory of "how we ship" — one file, reused by every project that imports it.
Net: AGENTS.md remembers the language, reflect remembers the last failure, the handoff remembers the last step, and methods remember how your team works — so you don't have to.

18 · Best practices & gotchas reference

Each row is one decision — the ✓ Do and the ✗ Don't side by side.

✓ DO
✗ DON'T
Verification — "can't fake done"
✓Give a real done when — a test, scanner, or script the world can run.
✗End because the model said "done". An unverified loop is just a prompt.
Thrash guard
✓Put after N tries: stop and warn … on every loop that reflects on failure.
✗Lean on the hard cap of 25 — it's a backstop, not a plan. No warning, wasted cycles.
Context scope
✓Point look at: at the relevant module and the last failure.
✗Leave context open and let the agent wander the whole repo.
Risky actions
✓Gate them — ask me before migrations or pushes; a human approves before deploys.
✗allow … automatically for destructive classes. (Pushing to main is refused regardless.)
Composition
✓Split big work: flow across files, for each per item — keep each file focused.
✗Cram an epic into one giant loop you can't read or reuse.
Cross-step state
✓Persist big artifacts to disk (a PRD, sprint.yaml) and look at: them next step.
✗Expect the handoff to carry structured data — it's a text summary, by design.
Transitions
✓Trust the fixed precedence: blocked > attempts > pass > fail.
✗Assume rules fire in the order you wrote them. They don't.
Discovery & human-only loops
✓Point done when at a validator of the artifact the loop produces.
✗Assume a loop with no done when ends when the chat goes quiet — give it a human gate or a when …: stop.

A loop that follows every "Do" above — real check, scoped context, gated risk, thrash guard:

loop "harden the upload endpoint":
  goal: uploads reject oversized and wrong-type files
  done when "pnpm test upload" passes        # real check — can't fake done
  look at: the upload code, and the last failure   # scoped context
  ask me before migrations or pushes          # gate risky classes
  each cycle: plan, then act, then observe
  when it fails: reflect, then plan again
  after 6 tries: stop and warn "stuck"        # thrash guard

That's the whole language. Five decisions, five run phases, and a handful of ways to compose them — loop → pipeline → flow → for each. Now go edit a loop.
